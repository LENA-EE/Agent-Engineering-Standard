# 🧭 AI Engineering Standard (AES)
![AES Workflow](aes_v12_workflow.png)
> **Стандарт разработки ПО с ИИ-агентами.**

**AI Engineering Standard (AES)** — структурированная методология взаимодействия человека и ИИ в разработке программного обеспечения.

AES превращает ИИ из генератора кода в **предсказуемого инженерного партнёра**.

---

**От промптов → к инженерии.**

---

## 📜 Обязательные контракты

AES требует два файла:

* `PROJECT_CONSTITUTION.md` — определяет ЧТО строим
* `CLAUDE.md` — определяет КАК агент строит

Оба контракта обязательны для соответствия AES.

## 🌍 Зачем разработке стандарт

Разработка с ИИ без стандарта часто приводит к:

* архитектурному хаосу
* потере контекста между сессиями
* непредсказуемым регрессиям
* несогласованным реализациям
* бесконечным переписываниям

AES вводит **контракты исполнения, инженерные правила и операционные границы**, делая разработку с ИИ контролируемой и масштабируемой.

---

## 🧠 Ключевой принцип

> ИИ выполняет решения — он их не изобретает.

Чёткое разделение ответственности:

| Роль | Ответственность |
| --- | --- |
| Человек | Видение продукта, архитектура, приоритеты |
| ИИ-агент | Анализ, реализация, валидация |

---

## 🧩 Спецификация AES

AES состоит из **двух обязательных контрактов**.

---

### 1️⃣ PROJECT_CONSTITUTION.md — Продуктовый контракт

Определяет:

* скоуп проекта и не-цели
* архитектурные решения
* технологический стек
* доменную модель
* границы безопасности
* стратегию деплоя
* критерии готовности

Отвечает на вопрос:

➡️ **Что мы строим?**

---

### 2️⃣ CLAUDE.md — Контракт исполнения

Определяет:

* роль и границы ИИ-агента
* workflow разработки (конечный автомат)
* инженерные ограничения
* запрещённые практики
* приоритеты разработки
* систему команд

Отвечает на вопрос:

➡️ **Как агент должен строить?**

---

## ⚙️ Стандартные команды агента

| Команда | Поведение |
| --- | --- |
| `/constitution` | Создать конституцию проекта |
| `/explore` | Анализировать репозиторий |
| `/spec` | Создать спецификацию фичи |
| `/plan` | Сгенерировать план реализации |
| `/status` | Показать статус проекта |
| `/fix` | Найти и исправить ошибки |
| `/review` | Провести код-ревью |
| `/deploy` | Подготовить к деплою |

ИИ **никогда не начинает реализацию без утверждённого плана**.

## 🔄 Жизненный цикл разработки AES

### Фаза 1 — Исследование (Discovery)

* анализ проекта
* создание конституции
* валидация контекста

### Фаза 2 — Планирование (Planning)

* спецификация (spec.md)
* пошаговый план (plan.md)
* идентификация рисков
* утверждение человеком

### Фаза 3 — Реализация (Execution)

* итеративная реализация
* соответствие архитектуре
* непрерывная валидация

### Фаза 4 — Поставка (Delivery)

* проверка стабильности
* очистка от дебаг-артефактов
* готовность к production

## 🧱 Инженерные требования

### Качество кода

* использовать типизацию, доступную в стеке проекта
* один файл — одна ответственность
* новые файлы: компактные и сфокусированные
* осмысленные имена

### Архитектура

* разделение слоёв
* изоляция зависимостей
* конфигурация через переменные окружения

### Надёжность

* защита внешних вызовов
* graceful degradation
* структурированное логирование
* восстановление после сбоев

### UX (базовый уровень)

* состояние загрузки
* пустое состояние
* состояние ошибки
* подтверждение деструктивных действий

---

## ⛔ Нарушения AES

❌ Подавление ошибок типизации и линтера
❌ Захардкоженная конфигурация
❌ Секреты в исходном коде (токены, пароли, API-ключи)
❌ Дебаг-логи в продакшне
❌ Мёртвый или закомментированный код
❌ Пустые catch-блоки
❌ Дублирование логики
❌ TODO без описания
❌ Магические константы

---

## 🛡️ Защищённые ресурсы

AES определяет жёсткую границу вокруг ресурсов, которые агент **никогда** не модифицирует без явного запроса:

* Системные конфиги — `.gitconfig`, `.ssh`, `.env`, профили оболочки
* База данных — изменение схемы, модификация данных, чужие миграции
* Инфраструктура — CI/CD, Docker, lock-файлы, конфиги сборки
* Git — политики веток, хуки, remote, identity пользователя
* Креденшалы — API-ключи, токены, конфиги vault, сертификаты

Если задача косвенно требует изменения защищённого ресурса — агент **останавливается и спрашивает**.

## ✅ Уровни соответствия AES

| Уровень | Описание | Необходимые артефакты |
| --- | --- | --- |
| AES-L1 | Конституция определена | PROJECT_CONSTITUTION.md существует и заполнен |
| AES-L2 | Спецификация существует | specs/ содержит ≥1 утверждённую спеку с планом |
| AES-L3 | Архитектура enforced | CLAUDE.md настроен, 0 нарушений в последнем PR |
| AES-L4 | Production Ready | CI проверяет AES violations автоматически |

Проекты могут декларировать:

```
AES: L3 Compliant
```
